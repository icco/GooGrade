Vision and Scope Background

  • Managing, calculating, and monitoring grades is time consuming for teachers!
  • Students want to monitor their progress in a course and plan ahead to obtain the best grades they can.
  • Students and teachers want freedom to do all of these things from home or the most convenient location with instant grade updates and feedback for students.

Business Opportunity

  • Teachers want to spend more time teaching rather than doing administration.
  • They want a quality grading tool that will save time by keeping track of assignments and students.
  • GooGrade will be competing with other tools of similar nature (especially the ones in our CPE308 class).
  • GooGrade has more features than most of other tools, and is basically free to use.

Competitive Analysis

Business Objectives

  • Save Time!
  • Create a free web based grade management application

Success Criteria

  • Measured by positive and negative feedback through a survey given at 3 months and 12 months of initial release.
  • If two or more Cal Poly Computer Science teachers choose to use GooGrade over any other application that already exists or developed in this class.

Risks

  • A University may be under contract to use one of GooGrade's competitior's applications.
  • Other tools exist too!

Noteworthy Features

  • Attendance management for courses.
  • Desired grade calculator.

Architecture Overview

  • MVC
    Model, View, Controller
  • Model
    Defines and pulls data from database (Hibernate?)(Serializable Java Class (JavaBean))
  • View
    JSP. Uses basic HTML and specialized tags to display data.
  • Controller
    Defines how to work with data.(Serverlet)

Entity Relationships

Entity/Class Summary

This is just a summary, main class diagram afterwards.

Here we go...

Account

  • An Account is the foundation of our three user types: Teacher, Teacher Assistant and Student.
  • It is where information for all three of the user types originate.
    • email addresses
    • names
    • user names

Teacher

  • Administrator
  • Owner of the System
  • Access to all information of any account associated with their courses

TeacherAssistant

  • The most versatile user type
  • Ranges from mere slave to almighty administrator
  • Available features completely dependent on set permissions

Student

  • A student's own account
  • Limited access to other students information

Permissions

  • Optional user features
  • Set of booleans
  • Always TRUE for Teacher
  • Varies for Student and Teacher Assistant

EmailAddress & Password

  • Contain methods to ensure that they are valid.

    • Course

    • A Course is an important storage class for a list of Students, Teacher Assistants, Grading Rules and other details about the course.
    • Importantly, it houses methods that manage the list of Students.

    Assignment

    • The Assignment class contains the information about a particular Assignment, including the points it is worth, the Due Date and its Description.
    • An Assignment can be one of several categories: Test, Quiz, Homework, Final, Participation, and Other.

    GradingRules

    • Variable grade boundaries
    • Single set for all assignments in a course

    File

    • Each instance of the File class represents one file uploaded by a user onto the system.
    • The class keeps tabs on who the file is from.
      • For instance, homework uploaded by a Student would be put up for download for a Teacher

    Announcement

    • The Announcement class represents the messages a Teacher wishes to display to his Students.
    • The class stores the fields for the Title and Body of the text the Teacher has entered.
    • In addition, it also will collect the date and time of modification and store it as well.

    Email

    • The Email class contains all the fields needed for an Email to be sent out to Students in a course.
    • Emails are not stored in the Database in anyway
