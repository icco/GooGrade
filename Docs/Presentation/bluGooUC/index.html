Use Case Summary

Actor: Teacher, Teacher Assistant w/ ManageUser Permission in their course, Student w/ ManageUser Permission can edit their own data.
Priority: High.
Frequency of Use: Once a Quarter

Description

The Actor can edit an Account from the system. The System will notify the Account holder about any changes made by Email.

Pre-conditions

  • The Actor must be authenticated by the system.

Post-conditions

  • Any other Account must be unaffected by the editing of an Account.
  • Password must comply with security requirement SE-4. (See section 5.3 in SRS document)

Normal Flow

  1. The Actor selects an Account for editing.
  2. The System displays the selected Account.
  3. The Actor selects what Account data to edit.
  4. The Actor edits selected Account data.
  5. The Actor selects to store any changes.
  6. The System validates edited Account data.
  7. The System stores edited Account data.
  8. The System notifies Account holder of any changed data.

Use Case Summary Cont.

Alternate Flow

Alt Flow 1: At step 5 of Normal Flow:
  1. The Actor selects to cancel the edit.
  2. The System returns to step 1 without altering any Account.
Alt Flow 2: At step 6 of Normal Flow:
If edited Password does not comply with SE-4 (See section 5.3 in SRS)
  1. The System displays an error message.
  2. The System returns to step 4 without altering any Account.
Alt Flow 3: At step 6 of Normal Flow:
If edited EmailAddress is not a valid email address.
  1. The System displays an error massage.
  2. The System returns to step 4 without altering any Account.

Use Case Summary Cont.

Functional Requirements

  1. When Actor selects "Edit User," the System displays a list of Accounts.
  2. When Actor selects "Submit", the system prompts the user to confirm the edits. If the user selects "Accept", the Account changes will be saved. If the user selects "Cancel", the system will return to the "Edit User" display.
  3. If no Accounts exist, the System displays "No Accounts Exist" and redirects the user to "Add User".
  4. If the Username specified already exists, the system displays "Username Already Exists" and Actor must correct the error to continue.
  5. If the EmailAddress is already in use, system displays "Email Address Already Exists" and Actor must correct the error to continue.
  6. If EmailAddress isn't valid, the System displays "The Email Address is Invalid" and Actor must correct the error to continue.
  7. If EmailAddress is altered, both the old and the new address will be notified of the changes.
  8. If Password does not comply with SE-4 (see Section 5.3 in SRS), the system displays "The Password is Not Secure Enough" and Actor must correct the error to continue.
  9. The System will confirm changes to the Actor after the changes are successfully stored.
  10. If the System fails to store changes the System displays "Could not store data".
SE-4: User Password must contain at least 6 characters and must include the following: at least 1 number, at least 1 capital character and the Password may not be the same as the Username.

Test Case Summary

Normal Flow

  1. Select an Account for editing.
  2. Observe that the selected Account is displayed.
  3. Select a data field to edit and change the data. Also change EmailAddress.
  4. Select to store all changes.
  5. Observe that any changed data is confirmed with a notification.
  6. Check both the old and new Email Addresses and observe that notifications were sent.

Alternate Steps 3

  1. Select and edit an Account.
  2. Choose to "Cancel" the edit.
  3. Observe that "Edit User" is displayed.

Alternate Steps 1

  1. Administratively ensure that Account data cannot be changedi.
  2. Attempt to select and edit an Account.
  3. Observe that an error is displayed.

Alternate Steps 4

  1. Ensure that no Users exist.
  2. Observe that an error is displayed.

Alternate Steps 2

  1. Choose to edit an Account.
  2. Change the password to one that does not comply with SE-4 (see Section 5.3 in SRS).
  3. Observe that an error is displayed.
